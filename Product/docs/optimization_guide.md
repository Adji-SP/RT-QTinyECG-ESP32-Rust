# Optimization Guide

This guide explains how to improve model accuracy, quantization alignment, and embedded resource usage.

## Current Baseline

After aligning the training and firmware feature pipelines, the current dry-run baseline is:

| Metric | Value |
|---|---:|
| PC float32 accuracy | 96.5% |
| ESP32/int8 dry-run accuracy | 95.4% |
| Quantization delta | 1.05% |
| Agreement | 98.9% |

This is already within the project targets.

## Recommended Workflow

```text
1. Retrain full model
2. Quantize weights
3. Export Rust weights
4. Run dry-run evaluation
5. Run hardware UART-feed evaluation
6. Compare metrics
7. Fine-tune only if metrics are outside target
```

## Decision Tree

```text
Run compare_models.py
    |
    +-- Quantization delta > 5%?
    |       Check feature extraction, scaler use, and weight layout first.
    |
    +-- ESP32/int8 accuracy < 85%?
    |       Inspect comparison_report.csv and confusion matrix.
    |
    +-- Agreement < 95%?
    |       Check dry-run vs hardware behavior.
    |
    +-- Recall too low?
    |       Add abnormal patterns or tune model threshold/data.
    |
    +-- Precision too low?
            Add normal/noise examples or reduce over-detection.
```

## Full Retraining

Use full retraining when:

- Synthetic data generation changes.
- Feature extraction changes.
- Firmware inference logic changes.
- `model_weights.rs` is stale.

```bat
py python\train_simple_model.py
py python\quantize_weights.py
py python\export_rust_weights.py
```

The training script uses the same firmware-compatible feature vector used by `inference.rs`.

## Fine-Tuning

Fine-tuning uses disagreement samples from `comparison_report.csv`.

```bat
py python\compare_models.py --save-disagreements data\disagreements.csv
py python\fine_tune_model.py --extra-data data\disagreements.csv --epochs 300 --augment-factor 5
```

Then re-export if the fine-tuned model is accepted:

```bat
py python\quantize_weights.py
py python\export_rust_weights.py
```

Current safety behavior:

- `fine_tune_model.py` compares the new model against the previous model.
- It keeps the old `model.pkl` if the fine-tuned model is worse.
- This is important because disagreement samples can be small and biased.

## What `auto_disagreements.csv` Does

`data/auto_disagreements.csv` is generated by `fine_tune_model.py` when it finds disagreements in `comparison_report.csv`.

It contains:

```text
index,gt_label,pc_prediction,esp32_prediction
```

It is an intermediate helper file, not an inference input. It can be deleted and regenerated.

## Architecture Experiments

Current firmware supports 5 -> 8 -> 1 directly. If you change architecture, update both training and firmware inference.

| Architecture | Notes |
|---|---|
| 5 -> 4 -> 1 | Smaller, lower capacity |
| 5 -> 8 -> 1 | Current supported default |
| 5 -> 16 -> 1 | More capacity, requires firmware updates |
| 5 -> 8 -> 8 -> 1 | Requires multi-layer firmware update |

For this codebase, keep `5 -> 8 -> 1` unless you also edit `inference.rs`, `model_weights.rs` generation, and evaluation simulation.

## Quantization Experiments

Current quantization:

```text
int8 weights
i32 biases
per-layer symmetric scale
```

Good target:

```text
PC accuracy - int8 accuracy < 3%
```

If the delta grows:

1. Confirm training uses firmware-compatible features.
2. Confirm `export_rust_weights.py` transposes W1 and W2 correctly.
3. Confirm dry-run simulation matches `inference.rs`.
4. Only then consider changing quantization strategy.

## Signal Parameters

| Parameter | File | Current |
|---|---|---:|
| Sample rate | `main.rs` | 250 Hz |
| Ring buffer | `main.rs` | 128 samples |
| Moving average | `main.rs` | 8 samples |

Tradeoffs:

| Change | Benefit | Cost |
|---|---|---|
| Smaller ring buffer | Lower latency | Less context, possible lower accuracy |
| Larger ring buffer | More context | Higher initial latency |
| Smaller filter window | Faster response | More noise |
| Larger filter window | Smoother signal | More lag |
